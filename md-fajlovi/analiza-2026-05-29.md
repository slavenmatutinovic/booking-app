## 🔴 KRITIČNI BAGOVI (mogu prouzrokovati grešku u produkciji)

### BUG-01 — Dvostruka definicija `JwtPayload` interfejsa u `authMiddleware.ts`

**Fajl:** `backend/src/middleware/authMiddleware.ts`, linije 48 i 97

Interfejs `JwtPayload` je definisan **dva puta** u istom fajlu:

```typescript
// Linija 48
interface JwtPayload {
  userId: string;
  role: 'ADMIN' | 'VIEWER';
  tokenVersion: number;
}

// Linija 97 — DUPLIKAT
interface JwtPayload {
  userId: string;
  role: 'ADMIN' | 'VIEWER';
  tokenVersion: number; // Dodajemo tvoj tokenVersion u tipove
}
```

TypeScript u `strict` modu ovo može prihvatiti jer su identični, ali je suvišno i konfuzno. Ako se jedan izmeni a drugi ne, nastaje nekonzistentnost.

**Rešenje:** Ukloniti drugu definiciju (linija 97) i zadržati samo prvu.

### BUG-02 — Dvostruko dodeljivanje `req.user` u `requireAuth`

**Fajl:** `backend/src/middleware/authMiddleware.ts`, linije 147–153

```typescript
// Linija 147–149 — prva dodela (pravilno filtrira polja)
req.user = {
  userId: payload.userId,
  role: payload.role as 'ADMIN' | 'VIEWER',
};

// Linija 153 — druga dodela ODMAH PREPISUJE prvu (dodaje tokenVersion u req.user!)
req.user = payload;
```

Druga dodela (`req.user = payload`) **prepisuje prvu** i ubacuje `tokenVersion` u `req.user`, što nije deklarirano u Express tipu i može prouzrokovati TypeScript greške ili curenje podataka. Tip `req.user` je deklarisan bez `tokenVersion`, ali payload ga sadrži.

**Rešenje:** Ukloniti liniju `req.user = payload;` — ostaje samo prva dodela.

### BUG-03 — `dragValid` nikad nije `false` — vizuelna validacija drag-and-drop ne radi

**Fajl:** `frontend/src/hooks/useDragDrop.ts`, linija 21

```typescript
const [dragValid] = useState<boolean>(true); // Nikad se ne menja!
```

`dragValid` je uvek `true` jer `setDragValid` nikada nije pozvano. Korisnik nikada ne vidi vizuelni indikator da prevlači rezervaciju na zauzet termin — traka ostaje zelena čak i kada bi konflikt bio odbijen od servera.

**Rešenje:** Implementirati proveru konflikta tokom drag-a koristeći `occupiedSet` i ažurirati `dragValid` u `handleGlobalMouseMove`.

### BUG-05 — `calendarActions.ts` koristi `React.Dispatch` bez importa React tipa

**Fajl:** `frontend/src/hooks/calendarActions.ts`, linije 60, 174, 212

Fajl eksplicitno napominje da "nema React importa", ali koristi `React.Dispatch<React.SetStateAction<...>>` u tipovima parametara. Ovo funkcioniše samo uz `import type React from 'react'` ili `import type { Dispatch, SetStateAction } from 'react'`.

```typescript
setBookings: React.Dispatch<React.SetStateAction<FrontendBooking[]>>;
```

**Rešenje:** Dodati `import type { Dispatch, SetStateAction } from 'react';` i koristiti `Dispatch<SetStateAction<...>>`.

### BUG-06 — Brisanje rezervacije bez dijaloga za potvrdu

**Fajl:** `frontend/src/components/BookingBar.tsx`

Pritiskom desnog klika na rezervaciju (context menu) ili klikom na X dugme, rezervacija se **odmah briše** bez ikakve potvrde korisnika. U `AdminDashboard.tsx` za odbijanje zahteva postoji `confirm()`, ali za brisanje rezervacije u kalendaru — ne postoji.

```typescript
const handleContextMenu = (e: React.MouseEvent<HTMLDivElement>) => {
  if (!canEdit || isDeleting) return;
  e.preventDefault();
  deleteBooking(b.id); // ← Direktno, bez potvrde!
};
```

### BUG-07 — `useCalendarData` fetchuje dvaput na svakom renderu zbog `dayW` u dependency array-u

**Fajl:** `frontend/src/hooks/useCalendarData.ts`, linija 167

```typescript
}, [startDate, dayW]); // dayW se menja pri svakom resize prozora!
```

`dayW` (širina jednog dana) se računa dinamički na osnovu veličine prozora. Svaki resize prozora trigguje novi API poziv za apartmane i rezervacije. Logovi potvrđuju problem — inicijalno učitavanje pravi 4+ API poziva.

**Rešenje:** Ukloniti `dayW` iz dependency array-a (koristi se samo u `calcBookingStyle` koji je van hooka) ili koristiti `useRef` za debouncing.

## 🟠 SREDNJE-KRITIČNI BAGOVI

---

### BUG-08 — `PENDING_EMAIL` status postoji u šemi ali se nikad ne koristi

**Fajlovi:** `prisma/schema.prisma`, `cleanupCron.ts`

`RequestStatus` enum ima `PENDING_EMAIL` vrednost, a komentari u kodu pominju dvostepeni tok (prvo email verifikacija, pa odobravanje). Međutim, sistem **direktno postavlja** `PENDING_APPROVAL` — `PENDING_EMAIL` nikad nije dodeljen:

```typescript
const pendingStatus: RequestStatus = 'PENDING_APPROVAL'; // PENDING_EMAIL se ne koristi
```

Ovo znači da postoji neuImplementiran tok, a cron čisti i `PENDING_EMAIL` zapise koji nikad ne nastaju. Šema i kod su nekonzistentni.

---

### BUG-09 — Keš u `bookings.controller.ts` NE koristi `CACHE_KEYS` konstantu

**Fajlovi:** `backend/src/utils/cache.ts` i `backend/src/controllers/bookings.controller.ts`

`cache.ts` definiše centralizovanu konstantu:

```typescript
BOOKINGS: (month?: string, aptId?: string) => `bookings:${month ?? 'all'}:${aptId ?? 'all'}`,
```

Ali kontroler ručno gradi isti ključ lokalno:

```typescript
const cacheKey = `bookings:${month || 'all'}:${apartmentId || 'all'}`;
```

Razlika: `??` vs `||` — ovo može dovesti do različitih ključeva ako je vrednost `0` ili prazan string. Konstantu iz `cache.ts` treba koristiti svuda.

---

### BUG-10 — Ruta `/requests/count` definisana POSLE `/:id` — rizik od route konflikta

**Fajl:** `backend/src/routes/bookingsRoutes.ts`

```
router.delete('/:id', ...)          // Linija 155 — dinamička ruta
router.get('/requests/pending', ...) // Linija 165
router.patch('/requests/:id/reject', ...) // Linija 183
router.get('/requests/count', ...)   // Linija 185 ← Ovaj ne bi bio zahvaćen DELETE /:id ali redosled treba biti konzistentan
```

Sve `/requests/*` rute trebaju biti definisane **pre** `/:id` ruta da bi Express ispravno parsirao putanje. Express rešava ovo interno (statičke putanje imaju prioritet nad dinamičkim), ali redosled može zbunjivati i treba biti eksplicitan.

### BUG-11 — Excel backup se generiše za svaku izmenu bez rate limitinga

**Fajl:** `backend/src/controllers/bookings.controller.ts`

```typescript
// Fire & forget — poziva se posle SVAKOG create/update/delete
generateBookingExcel(`Izmenjena rezervacija: ${id}`);
```

Ako admin brzo pomiče rezervacije drag-and-drop-om (što logovi potvrđuju — dve izmene u sekundi), Excel fajl se generiše za svaki pojedinačni `PATCH`. To može akumulirati desetine fajlova u minutu i opteretiti disk i I/O.

**Rešenje:** Debounce mehanizam (npr. čekati 5 sekundi od poslednje izmene pre generisanja).

### BUG-16 — `React.StrictMode` uzrokuje dupli API poziv u development modu

**Fajl:** `frontend/src/main.tsx`

`React.StrictMode` namerno dvaput pokreće efekte u development modu. Kombinovano sa `useEffect([startDate, dayW])`, svaki mount pravi **4 API poziva** (2×apartments + 2×bookings) što je vidljivo u logovima. Ovo nije greška u produkciji, ali otežava debugging i maskira stvarni broj zahteva.

### BUG-17 — `exposedHeaders: ['set-cookie']` u CORS konfiguraciji je nepotreban

**Fajl:** `backend/src/server.ts`, linija 95

```typescript
exposedHeaders: ['set-cookie'],
```

`Set-Cookie` zaglavlje nikad nije direktno čitano JavaScript-om (HttpOnly kolačić). Ovaj header nije potreban u CORS konfiguraciji i može biti zbunjujući.

### BUG-18 — Nedostaje sanitizacija HTML entiteta u email template-ima

**Fajl:** `backend/src/utils/emailService.ts`

### BUG-20 — `calendarActions.ts` pravi konflikt proveru samo lokalno, ne i pri drag-and-drop

`executeCreateBooking` proverava konflikt pre kreiranja, ali `executeMoveBooking` (drag-and-drop) **nema lokalnu proveru konflikta**. Korisnik može prevući rezervaciju na zauzet termin — server će odbiti, ali klijent prikazuje pomerenu traku sve dok server ne odgovori.

## 🔵 PREDLOZI ZA POBOLJŠANJE

---

### POBOLJŠANJE-01 — Dodati WebSocket / Server-Sent Events za real-time ažuriranje

Trenutno, ako dva admina rade istovremeno, moraju ručno da osvežavaju stranicu da vide promene drugog. Uvođenje SSE (Server-Sent Events) za invalidaciju keša i push notifikacija bi drastično poboljšalo iskustvo u višekorisničkom okruženju.

---

### POBOLJŠANJE-02 — Pagination u kalendaru nije implementiran na frontendu

Backend podržava cursor-based paginaciju (`nextCursor`), ali frontend nikad ne koristi `nextCursor` — uvek učitava prvu stranicu. Za korisnike sa stotinama rezervacija, calendar bi trebao implementirati "load more" ili automatski učitavati naredne stranice.

---

### POBOLJŠANJE-03 — AdminDashboard nema navigacionu traku / link nazad na kalendar

**Fajl:** `frontend/src/components/AdminDashboard.tsx`

`/admin/requests` ruta nema link nazad na `/calendar`. Admin mora ručno menjati URL. Potrebna je osnovna navigacija.

---

### POBOLJŠANJE-04 — Centralizovati logiku invalidacije keša u `cache.ts`

Invalidacija keša je kopirana na tri mesta u `bookings.controller.ts` (create, update, delete). Treba je izvući u utility funkciju:

```typescript
// U cache.ts
export function invalidateBookingCache() {
  const keys = appCache.keys().filter((k) => k.startsWith('bookings:'));
  keys.forEach((k) => appCache.del(k));
}
```

---

### POBOLJŠANJE-05 — Konfuzni mix `alert()` i `confirm()` — zaameniti React UI komponentom

**Fajlovi:** `calendarActions.ts`, `AdminDashboard.tsx`, `useDragDrop.ts`

Aplikacija koristi nativne `alert()`, `confirm()` i `console.warn()` za korisničke povratne informacije. Ovo ne može biti stilizovano, ne podržava lokalizaciju i ne integriše se sa aplikacijom. Preporučuje se `toast` notifikacioni sistem (npr. `react-hot-toast` ili custom komponentna).

---

### POBOLJŠANJE-06 — Log fajl od 1.8MB je u repozitorijumu

**Fajl:** `backend/aplikacija.log` (1.8MB)

`.gitignore` sadrži `*.log` ali log fajl je uključen u ZIP arhivu. Log sadrži podatke o korisničkim sesijama (userId, email URL-ovi). Treba dodati rotaciju logova (npr. `pino-roll`) i eksplicitno isključiti log fajlove iz distribucije.

---

### POBOLJŠANJE-07 — Nema retry mehanizma za email slanje

**Fajl:** `backend/src/utils/emailService.ts`

Emailovi se šalju "fire and forget" bez retry logike. Ako SMTP server privremeno nije dostupan, email se gubi bez traga. Preporučuje se jednostavan retry mehanizam (3 pokušaja sa eksponencijalnim čekanjem) ili korišćenje email queue sistema.

---

### POBOLJŠANJE-08 — `useCalendarData` hook je prevelik i ima previše odgovornosti

**Fajl:** `frontend/src/hooks/useCalendarData.ts` (250+ linija)

Hook istovremeno upravlja:

- Fetching-om podataka
- Formatovanjem rezervacija
- Računanjem statistika
- CRUD akcijama
- Logoutom

Preporučuje se razdvajanje na `useBookingsFetch`, `useBookingStats` i zadržavanje akcija u `calendarActions.ts`.

---

### POBOLJŠANJE-09 — Nedostaje podrška za mobilne uređaje (touch events)

**Fajlovi:** `useDragDrop.ts`, `BookingBar.tsx`

Drag-and-drop koristi isključivo `mousedown/mousemove/mouseup` evente. Na mobilnim uređajima ovi eventi ne rade — potrebni su `touchstart/touchmove/touchend`. Aplikacija je praktično neupotrebljiva na telefonu za admin operacije.

---

### POBOLJŠANJE-10 — Dodati `Zod` validaciju i za `updateBookingSchema` — provera prošlog datuma

**Fajl:** `backend/src/validators/booking.validator.ts`

`updateBookingSchema` nema proveru da su datumi u budućnosti. Admin može izmeniti rezervaciju da počne pre godinu dana bez ikakve validacije na validator nivou (samo `BOOKING_CONFLICT` bi je sprečio ako postoji preklapanje).

---

### POBOLJŠANJE-11 — Dodati `index` na `Apartment.name` za brže pretraživanje

**Fajl:** `backend/prisma/schema.prisma`

`name` polje je `@unique` (što automatski kreira B-tree index), ali nema eksplicitnog kompozitnog indeksa za joinove sa `Booking` tabelom. Ovo je OK za mali broj apartmana, ali treba dokumentovati.

---

### POBOLJŠANJE-12 — Nema zdravstvenog pregleda endpointa (`/api/health`)

**Fajl:** `backend/src/server.ts`

Postoji `/api/test` ruta, ali nema standardnog `/api/health` endpointa koji vraća status baze podataka i verziju aplikacije. Ovo je potrebno za monitoring i deployment health check-ove.

---

### POBOLJŠANJE-13 — `AdminDashboard` ne može da vidi `expiresAt` za zahteve

**Fajl:** `frontend/src/components/AdminDashboard.tsx`

Admin vidi: apartman, gost, kontakti, period. Ali ne vidi `expiresAt` — ne zna koliko vremena ima da odgovori na zahtev pre nego što automatski istekne.

---

### POBOLJŠANJE-14 — Nema `VIEWER` specifičnih ruta — uloga je definisana ali skoro beskorisna

**Fajlovi:** `shared/index.ts`, `bookingsRoutes.ts`

`VIEWER` rola je definisana u šemi i tokenu, ali jedina razlika od anonimnog gosta je da VIEWER vidi detalje gostiju u kalendaru. Nema VIEWER-specifičnih ruta ili funkcionalnosti. Treba ili proširiti VIEWER ulogu ili je ukloniti.

---

### POBOLJŠANJE-15 — Nedostaje HTTPS redirect u produkcijskoj konfiguraciji

**Fajl:** `backend/src/server.ts`

Ne postoji middleware koji forsira HTTPS u produkciji. HttpOnly kolačić sa `secure: true` neće biti poslat na HTTP zahteve, ali nema ni ekspicitnog redirecta.

## ✅ ŠTA RADI DOBRO

- Pessimistic locking sa `FOR UPDATE` u PostgreSQL transakcijama — sprečava race condition
- `tokenVersion` mehanizam za invalidaciju sesija pri logout-u — ispravno implementiran
- CORS konfiguracija sa eksplicitnom whitelist-om — bezbedna
- Zod validacija na svim ulaznim tačkama
- `optionalAuth` / `requireAuth` / `requireAdmin` razdvojenost — čist RBAC
- Optimistic update pattern sa rollback-om za drag-and-drop
- In-memory Mutex za sprečavanje race conditiona kod zahteva gostiju
- Cursor-based paginacija na backendu (čak i ako frontend ne koristi sve)
- Cron za automatsko čišćenje isteklih zahteva
- Excel backup posle svake mutacije (čak i ako treba debouncing)
- Helmet + rate limiter + compression — solidan sigurnosni stack
- ErrorBoundary na root nivou aplikacije
- `shared/index.ts` kao jedini izvor istine za zajedničke tipove
