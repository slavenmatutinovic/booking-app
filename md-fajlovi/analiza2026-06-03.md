## 3. Bezbednosni propusti

### 🔴 SEC-01: `requireAuth` koristi hardkodovani fallback JWT secret

**Fajl:** `backend/src/middleware/authMiddleware.ts` (linija 104)

**Problem:**

```typescript
// requireAuth
const JWT_SECRET = process.env.JWT_SECRET || 'fallback_secret';
const payload = jwt.verify(token, JWT_SECRET) as unknown as JwtPayload;
```

Ako `JWT_SECRET` nije definisan u `.env`, aplikacija tišim padom ne lupa — nastavlja da radi sa `'fallback_secret'`. Svako ko zna ovaj string može da generiše validan JWT token i dobije admin pristup.

**Ista linija u `optionalAuth` je ispravna:**

```typescript
// optionalAuth — ispravno koristi env.JWT_SECRET (validiran pri startu)
const payload = jwt.verify(token, env.JWT_SECRET) as JwtPayload;
```

Nekonzistentnost između dve funkcije u istom fajlu.

**Ispravka:**

```typescript
// requireAuth — linija 104
const payload = jwt.verify(token, env.JWT_SECRET) as unknown as JwtPayload;
```

`env.JWT_SECRET` dolazi iz `env.ts` koji pada pri startu ako secret nije validan (min 32 karaktera) — to je ispravno ponašanje.

## 4. Pricing sistem — analiza i nedostaci

### Arhitektura

```
Admin UI                     Backend                    Frontend kalkulacija
ApartmentRatesManager  →  POST /api/apartments/rates →  createApartmentRate (DB)
                                                             ↓
GET /api/apartments (bez rates!)                     ← rates nisu uključene
                                                             ↓
BookingModal → BookingPricePreview ← calculateClientDynamicPrice([], fallback=50€)
```

### Kompletna lista pricing propusta

| Br. | Propust | Ozbiljnost | Fajl |
|-----|---------|------------|------|
| P-1 | Ruta `/apartments/rates` je pogrešna (dupliran prefix) | 🔴 Kritično | `apartmentsRoutes.ts` |
| P-2 | `getApartments` ne uključuje `rates` u odgovor | 🔴 Kritično | `apartments.controller.ts` |
| P-3 | Nema `DELETE /rates/:id` endpointa | 🔴 Kritično | `apartmentsRoutes.ts` |
| P-4 | Nema `PATCH /rates/:id` endpointa za izmenu cene | 🟡 Važno | — |
| P-5 | Nema `GET /api/apartments/:id/rates` endpointa za izlistavane svih cena apartmana | 🟡 Važno | — |
| P-6 | `pricing.ts` importuje Prisma na frontendu (mrtav kod) | 🔴 Kritično | `frontend/utils/pricing.ts` |
| P-7 | Fallback cena (50€) nije konfigurisabilna ni po apartmanu ni globalno | 🟡 Važno | — |
| P-8 | `rate.validator.ts` dozvoljava `endDate === startDate` (nulta dužina) | 🟡 Važno | `rate.validator.ts` |
| P-9 | `BookingPricePreview` nema validaciju `totalNights === 0` pre renderinga | 🟢 Minor | `BookingPricePreview.tsx` |
| P-10 | Nema prikaza cene pri drag&drop pomeranju rezervacije | 🟢 Minor | `useDragDrop.ts` |
| P-11 | Cena se ne čuva uz rezervaciju (nigde u `Booking` modelu) — nemoguća revizija | 🔴 Kritično | `schema.prisma` |

### P-11 Detalj — Ozbiljan arhitekturalni propust

`Booking` model u Prisma šemi ne sadrži polje za ukupnu ili dnevnu cenu:

```prisma
model Booking {
  id          String   ...
  guest       String
  email       String
  startDate   DateTime
  endDate     DateTime
  status      BookingStatus
  // totalPrice  Decimal?  ← NEDOSTAJE
  // pricePerNight Decimal? ← NEDOSTAJE
}
```

Posledice:
- Excel backup ne sadrži finansijske podatke
- Promena sezonske cene retroaktivno "menja" istorijske rezervacije u obračunu
- Nema osnove za fakture ili finansijske izveštaje

**Preporučena migracija:**

```sql
ALTER TABLE "Booking" ADD COLUMN "totalPrice" DECIMAL(10,2);
ALTER TABLE "Booking" ADD COLUMN "priceSnapshot" JSONB;
-- priceSnapshot čuva dnevni breakdown u trenutku rezervacije
```

### 🟡 BUG-06: `occupiedSet` u `useCalendarData.ts` uključuje CANCELLED rezervacije

**Fajl:** `frontend/src/hooks/useCalendarData.ts`

```typescript
const occupiedSet = useMemo(() => {
```

`bookings` niz sadrži samo `CONFIRMED` rezervacije (backend filtrira). Ali ako se lokalno optimistično otkazuje rezervacija bez brisanja iz niza, ili ako dođe do inconsistency, ćelije mogu biti pogrešno prikazane kao "zauzete".

Dodatno: kada admin otkaže rezervaciju, `setBookings` u `executeDeleteBooking` verovatno uklanja iz niza, ali bi eksplicitni filter bio sigurniji:

```typescript
bookings.filter(b => b.status !== 'CANCELLED').forEach(...)
```

### 🟡 BUG-07: `cache invalidation` u `createApartment` dešava se POSLE `res.json()`

**Fajl:** `backend/src/controllers/apartments.controller.ts`

```typescript
res.status(201).json({ message: 'Apartman je uspešno kreiran', apartment });

// INVALIDACIJA KEŠA — briše se stari niz
```

U normalnom toku ovo radi jer Node.js nastavlja da izvršava kod posle `res.json()`. Ali:
1. Ako dođe do greške posle `res.json()`, keš neće biti invalidiran

**Ispravak redosled:**

```typescript
appCache.del(CACHE_KEYS.APARTMENTS);
res.status(201).json({ message: 'Apartman je uspešno kreiran', apartment });
```

### 📁 `backend/src/middleware/authMiddleware.ts`

**Ocena: 7/10** — Jedan kritičan security bug

**Problemi:**
1. **Kritično:** `process.env.JWT_SECRET || 'fallback_secret'` — treba `env.JWT_SECRET` (linija 104)
2. `optionalAuth` ne koristi keš za `tokenVersion` proveru (za razliku od `requireAuth`) — svaki zahtev ide u bazu
3. `req.user` tip proširuje globalni Express namespace što je antipattern — bolje koristiti lokalni `declare module`

**Predlog za `optionalAuth`:**

```typescript
// Dodati keš kao u requireAuth
const cacheKey = `user:session:${payload.userId}`;
let cachedSession = appCache.get<...>(cacheKey);
if (!cachedSession) {
```

## 7. Šta radi dobro

✅ **JWT sa `tokenVersion`** — Ispravna implementacija revokacije sesije. Logout zaista poništava token.

✅ **Pessimistic locking** — `FOR UPDATE` SQL brava u transakcijama za `createBooking`, `updateBooking`, `deleteBooking`. Sprečava race condition pri konkurentnim zahtevima.

✅ **Zod validacija** — Konzistentna upotreba Zod šema na svim API endpointima (osim jedne nedoslednosti u `createGuestRequestSchema`).

✅ **Soft delete** — Rezervacije se ne brišu, status se menja u `CANCELLED`. Istorija je sačuvana.

✅ **Email verifikacioni tok** — Dvostepeni sistem (PENDING_EMAIL → PENDING_APPROVAL → APPROVED/REJECTED) je dobro dizajniran. Mutex brava po apartmanu sprečava duplirane zahteve.

✅ **Cursor-based paginacija** — I za bookings API i za Excel backup.

✅ **`shared/index.ts`** — Single source of truth za tipove. Oba end-a importuju odavde.

✅ **Cache invalidacija** — `invalidateBookingCache()` se poziva na svakoj mutaciji.

✅ **GDPR filtriranje** — Backend maskuje `guest`, `email`, `phone` za neautorizovane korisnike.

✅ **`env.ts` fail-fast** — Aplikacija pada pri startu ako nedostaju obavezne env varijable.

✅ **`optionalAuth` middleware** — Elegantno rešenje za javne rute sa opcionalnim podacima.

✅ **`useDragDrop` hook** — Optimistično ažuriranje sa rollback-om pri grešci.

✅ **Cleanup cron** — Automatsko brisanje isteklih zahteva na svakom satu.
